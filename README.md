## 프로젝트 개요

라이브 클래스 수강 신청 멀티 스텝 폼을 구현한 프로젝트.
강의 목록에서 카테고리별로 강의를 탐색하고, 신청자 정보를 입력하고, 입력을 확인한 뒤 개인/단체 신청을 진행한다.

백엔드는 MSW로 모킹되어 있어 별도 서버 없이 전체 플로우를 그대로 확인할 수 있다.

## 기술 스택

| 영역        | 선택                     | 선택 이유                                                                                                                      |
| ----------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------ |
| 언어        | TypeScript               | 과제 필수 요건. 정적 타입으로 개발 안정성 향상 위함                                                                            |
| 프레임워크  | React 19                 | 과제 권장 사항, 라이브러리 생태계 풍부함                                                                                       |
| 빌드 도구   | Vite                     | 간단한 셋업, 빠른 HMR 지원으로 개발 생산성 높음                                                                                |
| 스타일링    | shadcn/ui + Tailwind CSS | 무거운 디자인 시스템 의존성 없이 컴포넌트 소스를 직접 소유(복사 방식)하므로 커스터마이즈가 자유로움. 접근성 기본 보장          |
| 라우팅      | React Router             | React SPA 라우팅의 사실상 표준으로 친숙하고 생태계가 풍부함                                                                    |
| 서버 상태   | TanStack Query           | 서버 통신에서 로딩 및 에러 상태 관련 보일러플레이트 코드를 생략하고 선언적으로 구현하기 위함                                   |
| 폼 관리     | React Hook Form          | ref 기반으로 리렌더링을 최소화하고, 검증이나 에러 등 폼 관리를 깔끔하게 처리하기 위함                                          |
| 스키마/검증 | Zod                      | validation 검증과 폼 타입을 단일 스키마에서 도출하여 관리 포인트를 줄이기 위함, React Hook Form과 resolver로 자연스럽게 연동됨 |
| API Mock    | MSW                      | 네트워크 레이어에서 요청을 가로채므로 실제 데이터 페칭 코드를 그대로 유지하면서 Mock/실서버 전환이 용이함                      |
| 린트/포맷   | ESLint + Prettier        | ESLint로 코드 패턴 규칙 위반을 정적으로 잡고 Prettier로 포맷을 자동화                                                          |

## 실행 방법

패키지 매니저는 `pnpm`을 사용한다.

```bash
pnpm install        # 의존성 설치 (최초 1회)
pnpm dev            # Vite 개발 서버 실행 (기본 http://localhost:5173)
pnpm build          # 린트 + 타입 체크 + 프로덕션 빌드
pnpm preview        # 빌드 결과물 미리보기
pnpm lint           # ESLint 실행
pnpm format         # Prettier로 전체 파일 포맷팅
```

API는 MSW로 모킹되어 있어 별도 백엔드 없이 `pnpm dev`만으로 전체 플로우를 확인할 수 있다. MSW 워커는 `public/mockServiceWorker.js`로 등록되어 있다.

## 프로젝트 구조 설명

```
src/
├── main.tsx              # 앱 엔트리. MSW 워커 시동 후 React 마운트
├── App.tsx               # 라우터 / QueryClientProvider 등 최상위 트리
├── components/           # 범용 컴포넌트 + shadcn/ui (src/components/ui)
├── hooks/                # 전역에서 재사용 가능한, 비즈니스 로직이 제거된 훅 (예: useLeaveGuard)
├── lib/                  # 프레임워크 비의존 유틸 (fetcher, queryClient, format 등)
├── queries/              # API 경로·요청 함수·TanStack Query 훅. MSW와 동일한 식별자를 공유
├── mocks/                # MSW 핸들러와 인메모리 데이터 싱글톤
├── types/                # 도메인 enum / 타입 (Category, Enrollment 등)
└── pages/
    └── enroll/
        ├── EnrollPage.tsx       # 멀티스텝 신청 폼 (단일 URL, 컴포넌트 상태로 스텝 관리)
        ├── EnrollDonePage.tsx   # 신청 완료 화면
        ├── _components/         # 이 페이지에서만 쓰이는 컴포넌트
        ├── _hooks/               # 이 페이지 전용 훅
        ├── _constants/           # 이 페이지 전용 상수 (에러 메시지 카피 등)
        └── _schema.ts            # Zod 스키마 (개인/단체 discriminatedUnion)
```

- `@/*` alias가 `src/*`에 매핑되어 있으므로 `src` 경계를 넘는 import는 항상 `@/...` 형태로 작성한다.
- 페이지 폴더 안의 `_components` / `_hooks` / `_constants` / `_schema`는 해당 페이지 내부에서만 쓰이는 파일을 유형별로 모아둔 것이다. 공용으로 끌어올릴 가치가 생기면 `src/components/`·`src/hooks/`로 이동한다.

## 요구사항 해석 및 가정

### 단체 신청 시 강의의 잔여석이 참가자 수만큼 남아있어야 함

명세에는 "정원 초과 시 `COURSE_FULL`"로만 기술되어 있어, 단체 신청에서 정원 체크의 기준이 모호하다. 잔여석이 1자리만 있으면 단체 신청 자체는 허용하고 가능한 인원만 등록하는 방식도 가능하지만, 신청 단위로 보면 "5명 단체로 신청했는데 2명만 등록됨" 같은 부분 성공 상태가 더 혼란스럽다고 판단했다.

- 따라서 `currentEnrollment + 참가자 수 > maxCapacity`이면 신청 전체를 `COURSE_FULL`로 거절한다. 단체는 all-or-nothing으로 처리한다.

### 한국 전화번호는 9~11자리를 모두 허용

명세에 "한국 전화번호 형식"으로만 기술되어 있어, 휴대폰(`010-1111-1111`)뿐 아니라 지역번호 유선(`02-123-4567`)과 구형 휴대폰(`011`, `016` 등) 입력까지 가정에 포함했다.

- Zod regex로 9 / 10 / 11자리를 모두 통과시키고, 입력 중에는 길이에 따라 하이픈 포맷팅을 자동 적용한다.

### 임시 저장은 "입력이 한 번이라도 발생한 이후"부터 시작

명세의 "새로고침 후에도 입력 데이터 복구"를 만족하되, 신청 페이지에 진입만 한 사용자가 의도치 않게 draft를 남기는 동작은 피했다.

- 첫 스텝에서 사용자가 어떤 필드도 건드리지 않았다면 자동 저장하지 않는다. 즉 빈 draft는 생성되지 않는다.
- 한 번이라도 변경이 일어난 이후에는 이후 스텝의 변경까지 모두 누적해 저장한다.

## 설계 결정과 이유

### TanStack Query 캐싱 전역 `staleTime: 1000 * 60`

전역 기본값은 보수적으로(=캐시를 좀 더 길게 신뢰) 잡고, 실시간성이 중요한 쿼리에서만 개별적으로 짧게 override하는 전략을 택했다. 캐싱 정책을 데이터 성격에 맞게 따로 다루는 편이 일반적이기 때문이다.

기본값을 `1000 * 60`(1분)으로 정한 근거는 다음과 같다.

- TanStack Query 공식 SSR 가이드에서 권장 예시로 사용하는 무난한 출발값이다.
- 너무 짧으면(`0`) 라우팅·포커스 전환마다 재요청이 터져 캐싱의 의미가 사라지고, 너무 길면 정적이지 않은 데이터까지 stale 노출 위험이 커진다.

### 강의 조회 쿼리는 `staleTime: 0`으로 override

`useCoursesQuery` / `useCourseQuery`는 개별적으로 `staleTime: 0`을 지정했다.

강의의 잔여석(`currentEnrollment` / `maxCapacity`)은 다른 사용자의 신청에 따라 실시간으로 변한다. 캐시된 값을 보여 주다가 신청 단계에서 `COURSE_FULL` 에러가 나면 사용자 경험이 떨어지므로, 화면 진입 시점마다 최신 데이터를 다시 가져오도록 설정했다.

### 멀티스텝 신청 폼은 단일 URL에서 진행

신청 폼의 3단계 진행을 `/enroll/step-2`, `/step-3`처럼 라우트로 쪼개지 않고 `/enroll` 한 라우트 안에서 컴포넌트 상태로만 관리한다.

- 멀티 스텝 폼 페이지는 항상 1스텝부터 직접 데이터를 쌓아 가며 진행해야 하는 페이지이므로, URL을 통해 외부에서 특정 스텝으로 직접 진입하는 동선을 차단했다.

### Suspense + ErrorBoundary로 로딩/에러 처리 선언화

강의 목록 화면의 로딩·에러 분기를 `isPending` / `isError`로 직접 다루는 명령형 방식 대신, `useSuspenseQuery` + `<Suspense>` + `<ErrorBoundary>` 조합으로 전환했다.

- **관심사 분리**: 데이터 컴포넌트는 "데이터가 있다"는 전제로만 작성되고, 로딩/에러는 트리 위쪽 경계가 담당한다. `data` 옵셔널 가드(`?.`, `?? []`)와 상태 분기가 사라져 본문이 도메인 로직에만 집중한다.
- **경계 단위로 UX 설계**: "어디까지를 함께 로딩/에러로 묶을지"를 컴포넌트 트리 위치로 표현할 수 있다. 카테고리 탭처럼 데이터와 무관한 UI는 경계 밖에 두어 로딩 중에도 유지되고, 그리드와 신청 바처럼 데이터에 의존하는 영역만 한 묶음으로 묶여 함께 fallback 처리된다.
- **재시도 자동화**: `resetKeys={[category]}`로 카테고리 전환 시 이전 에러가 자동 리셋되고, `resetErrorBoundary` 호출만으로 자식이 재마운트되며 Suspense fallback → 재요청 흐름이 자연스럽게 이어진다.

트레이드오프로 `enabled: false` 같은 조건부 비활성화나, "재시도 중에도 이전 에러 UI 유지" 같은 세밀한 상태 공존은 포기하게 된다. 이 화면에선 그런 요구가 없어 선언형의 단순함이 더 큰 이득이라 판단했다.

### 상태 관리 : Tanstack Query, React Hook Form, props

상태를 성격별로 나눠 각자에게 적합한 도구로 처리했다.

- **서버 상태**: TanStack Query. 강의 조회·신청 같은 원격 데이터의 캐싱·무효화·로딩/에러 처리를 일임한다.
- **폼 상태**: React Hook Form. 신청 폼의 입력값과 검증 상태는 RHF가 ref 기반으로 관리한다.
- **컴포넌트 간 지역 상태**: 부모 → 자식 props 전달로 해결했다. 깊이가 깊지 않아 Context API는 사용하지 않았다.

Redux / Zustand 같은 전역 상태 라이브러리는 도입하지 않았다. 이 과제에는 로그인 사용자 정보처럼 화면을 가로질러 공유해야 하는 진짜 전역 상태가 없고, 전역 상태 라이브러리가 필요할 만큼 리렌더링 부담이 심각하지 않기 때문이다.

### 공용 fetcher는 `fetch`를 얇게 감싼다

`axios` 같은 두꺼운 HTTP 클라이언트 대신 `fetch`를 얇게 감싼 공용 fetcher를 두고, 모든 호출이 이 함수를 통과한다.

- **의존성 최소화**: 별도 라이브러리 없이 표준 `fetch`만으로 충분하다.
- **에러 처리 일원화**: `.ok` 분기, JSON 파싱 실패, 서버가 내려주는 에러 코드(`COURSE_FULL` 등) 매핑을 한 곳에서 표준화된 에러로 변환한다. 호출부는 throw된 에러를 그대로 받아 TanStack Query의 `onError` / ErrorBoundary로 흘려보낸다.

### API 정의는 `src/queries/`에 모아 MSW와 공유

강의·신청 API의 경로와 요청 함수는 `src/queries/`에서 정의하고, MSW 핸들러와 TanStack Query 훅이 같은 식별자를 참조하도록 했다.

- Mock 핸들러와 실제 호출이 별도 문자열로 갈라지면 한쪽만 바뀌었을 때 발견이 늦어진다.
- "Mock/실서버 전환 용이성"이라는 MSW 채택 이유와 같은 방향의 구조 결정이다.

### 도메인 enum은 TS `enum` 대신 const object 패턴

`tsconfig.json`의 `erasableSyntaxOnly`로 TS `enum`을 사용할 수 없다. 따라서 `Category` 같은 도메인 enum은 `as const` 객체 + `Object.values`로 도출한 리터럴 유니온 + 라벨 맵을 같은 파일에서 함께 관리한다.

- **런타임 부산물 없음**: enum의 양방향 매핑 객체가 빌드 결과에 남지 않는다.
- **추가 비용 최소화**: 새 카테고리 추가 시 enum 객체에 한 줄 더할 때 리스트·라벨 맵·타입이 모두 자동 갱신된다.

### 폼 스키마는 Zod `discriminatedUnion`으로 개인/단체 분기

신청 폼은 개인/단체 두 흐름에서 필요한 필드가 다르므로, `type` 필드를 디스크리미네이터로 둔 `discriminatedUnion`으로 두 스키마를 합쳤다.

- **타입 좁히기**: `type === 'group'`일 때만 단체 전용 필드(인원수 등)가 검증되고, RHF / 컴포넌트 쪽에서도 분기마다 정확한 타입이 좁혀진다.
- **공통 검증 위치**: 약관 동의 같은 분기 공통 규칙은 각 분기 스키마 안에서 처리한다. 외부 `.refine`은 union을 풀어 디스크리미네이터의 이점을 잃게 만든다는 점을 검증 끝에 확인했다.

### Mock 데이터는 상수가 아닌 인메모리 싱글톤 store로 분리

`src/mocks/handlers.ts`가 직접 샘플 배열 상수를 참조하면 신청에 따른 잔여석 변화가 화면에 반영되지 않는다. 따라서 데이터를 별도 모듈의 살아 있는 인스턴스로 옮기고, 핸들러는 "요청 해석 → store 조작/조회"만 담당하게 했다.

- 잔여석 같은 동적 값이 실제 운영처럼 GET 요청마다 갱신된다.
- 도중 마감(`COURSE_FULL`) 같은 레이스 상황도 mock 단에서 자연스럽게 재현된다.

### 임시 저장은 watch + debounce + 언마운트 flush로 처리

폼 입력값을 매 키스트로크마다 localStorage에 기록하면 비용이 크고, 너무 길게 모으면 갑작스러운 이탈 시 마지막 입력을 잃는다. 그래서 RHF의 `watch` 구독으로 변경을 감지하고 debounce로 묶어 저장하되, 언마운트 시 미저장 상태로 남은 마지막 입력을 즉시 저장해 보존한다.

- 복원 시에는 localStorage에서 읽어온 JSON을 타입 단언으로 신뢰하지 않고 Zod `safeParse`로 검증한 뒤에만 폼에 주입한다. 스키마가 바뀌어 호환되지 않는 draft가 남아 있어도 안전하게 폐기된다.
- 참가자 인원 범위 같은 폼 제약과 default 객체는 `_schema`에 모아 검증 규칙과 UI 제약이 한 곳을 바라보도록 했다.

### 서버 에러는 인라인 배너로 표시

`COURSE_FULL` / `DUPLICATE_ENROLLMENT` / `INVALID_INPUT` 등 서버에서 내려주는 에러를 어떤 입력 때문에 막혔는지 컨텍스트와 함께 사용자가 어떤 행동을 해야 하는지 지침을 제공하기 위해 인라인 배너로 표시한다.

### 페이지 폴더 내부에 `_components` / `_hooks` / `_schema` 배치

공용 `src/components/`에는 shadcn/ui와 진짜 범용 컴포넌트만 두고, 한 페이지 안에서만 쓰이는 컴포넌트·훅·스키마는 `pages/<page>/_components/`, `_hooks/`, `_schema/`처럼 페이지 폴더 안에 둔다.

- **응집도**: 페이지 단위로 변경 영향 범위를 폴더 경계로 빠르게 파악할 수 있다.
- **라우팅 폴더와 시각적 구분**: `_` prefix는 "라우팅 후보가 아닌 내부 구현"임을 알리는 컨벤션으로 사용했다.

## 미구현 / 제약사항

## AI 활용 범위

모든 실질적 구현 단계에서 AI를 적극적으로 사용했다. 기능 요구사항을 바탕으로 코드 초안을 AI로 뽑고, 그 결과물을 기반으로 수정·리팩토링 명령을 내리면서 AI와 함께 의사결정하는 방식으로 작업했다.

- **설계·판단은 직접**: 기술 스택 선정, 폴더 구조, 멀티스텝 폼을 단일 URL로 유지하는 등 전체적인 설계 결정은 경험에 따라 직접 판단했다.
- **세부 구현은 AI**: 컴포넌트 골격, Zod 스키마 작성, MSW 핸들러, 폼 검증 처리 같은 반복적이고 결이 분명한 세부 구현은 AI의 힘을 빌려 초안을 만들고, 컨벤션·도메인 규칙에 맞게 다듬어 확정했다.
- **리뷰어로서의 AI**: 작성한 코드의 트레이드오프를 정리하거나 README의 설계 결정 문구를 다듬을 때도 AI를 사용했다. 다만 최종 문구와 결정은 모두 직접 검토 후 반영했다.
